# 序言

整理上一节代码

const Koa = require('koa')
const app = new Koa()
const Router = require('koa-router')
const router = new Router()

router.get('classic/latest', (ctx, next) => {
  ctx.body = { key: 'classic' }
})

app.use(router.routes())

app.listen(5000)

如果我们有很多的 api 接口，是否可以将一个个 api 接口都写在 app.js？（灵魂拷问）

答案是可以的，但是我们写代码不仅仅为了完成功能，我们也应该努力的写出好的代码，这有什么好处呢？

  • 别人阅读你的代码很方便
  • 代码利于维护
  • 好的代码有利于提高编程效率

如果将一个个 api 接口都写在 app.js 可以说是一种灾难，我们必须考虑把所有的路由按照一定的主题，进行拆分，将路由分别写在不同的文件里去。这个是我们要解决的问题

# 主题划分逻辑

每个人都有自己拆分主题的想法,基本的主题划分逻辑：我们根据数据的类型进行主题的划分。

如旧岛小样主题划分划分就分为期刊，书籍和点赞。

主题与主题之间不一定是绝对的孤立的，相互之间有的一定的联系

主题的划分直接关系到数据库的好坏。好的主题划分可以使得数据库设计变得十分简单，反之亦然。主题的划分是一个渐进式的过程，我们一般找到的是核心主题，还有一些小的主题附属在核心主题上面。这些小的附属主题在我们写代码的过程中，通过思考我们会一一找出。我们最关键的就是找到核心主题。

主题是一个抽象的概念，这有助于我们思考。 我们可以用Model 来对应主题这个概念。一个纯粹的前端开发者对于主题，数据库建表还有业务模型的设计可能会比较生疏。因为前端这块不太需要我们自己去思考这些主题或者是业务模型。大多数情况下后端给的 api 已经建立好了这些主题或者是业务模型。 这些主题或者是业务模型的设计能力需要在项目中磨砺，写多了就会了。

# 函数设计

函数设计对于异常的处理是有技巧的。

两种方案：

  • 一种是在函数内部判断出发生了异常，然后return false
  • 第二种 是在函数内部判断出发生了异常 throw new Error 很多人比较习惯第一种写法。

比如说在func3 里面发生异常了return false我们大多这么写。

function func1() {
  fun2()
}

function func2() {
  const r = func3()
  if (!r) {
    //逻辑
  }
}

function func3() {
  return false
}

我们可以看到throw error，一个ReferenceError异常,这是一个 node 内置的异常。

但是，我们在 try 里面都是处理我们自己的业务逻辑，比如说我们在里面调用的是微信的 api，那么微信的 api 可能返回一个错误信息。如果我们要抛出这个异常的话就要自定义一个异常。

加入说在 func3 里面抛出一个异常，那么在 func2 里面调用了 func3，我们也需要在 func2 里面 try…catch… 抛出异常，同理，在 func1 里面调用 func2 也是走一样的步骤。

    fun2()

    throw error
  }
}

function func2() {

    func3()

    throw error

function func3() {

    1 / a

这就是 js 中的异常链条，正常来说我们调用函数都需要使用 try…catch…。

这里说一下为什么很多书里面都不推荐return false 或者 null 来代替 throw new Error，这是因为 return false 或者 null 会丢失异常。比如说 func3 里面抛出了异常，func2 里面**try…catch…**到了之后没有 throw 这个异常，而是 return false 或者 null，那么 func1 就丢失了异常信息。

# 思考

实际上没有人没写一个函数都try…catch…，如果不想每调用一次函数都**try…catch…**怎么办呢?

第一种方法：你能确保有一个函数是万无一失的，永远都不会抛出异常。比如func3的代码是万无一失的，func2就不用try…catch…异常了。因为func3是我们自己写的，我们可以确保其万无一失。如果func3是一段别人的代码 如npm包，你不能保证别人的代码不会发生错误，不会抛出异常。所以在调用第三方库的时候应该去try…catch…。

第二种方法：全局异常处理。假如说func3 抛出了一个异常 在func 1.2.3我们都没有try…catch…。在所有的函数调用顶部，需要设置一种机制它可以监听到任何一种异常。这样就能保证在函数调用里面就不需要特别的try…catch…。

我们为什么要捕捉异常呢？因为我们需要告诉用户，同时我们在排查错误的时候需要这样的异常。

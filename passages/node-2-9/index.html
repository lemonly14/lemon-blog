为什么 koa 代码都会默认带上 async await？

app.use(async (ctx, next) => {
  console.log(1)
  next()
  console.log(2)
})

app.use((ctx, next) => {
  console.log(3)
  const axios = require('axios')
  const res = axios.get('http://7yue.pro')
  next()
  console.log(4)
})

整理前面写的两个中间件，实验洋葱模型,正常打印应该是 1 3 4 2

打印的结果：1 3 2 4

和预期不符合，正常按照洋葱模型的思路应该是 1 3 4 2 参考之前洋葱模型。

这是为什么呢？

关键原因是因为第二个中间件有 await 这个线程就阻塞了 这样的话 js 就去执行其他的流程 这个在上一节有提到 等到 await 取到了异步函数的值并赋值给 res 才会接着走下面的流程，这样就不能保证洋葱模型了。

那么我们该如何保证洋葱模型呢?

保证中间件按照洋葱模型执行的关键条件是每一个 next()前面都要加上 await

app.use(async (ctx, next) => {
  console.log(1)
  await next()
  console.log(2)
})

app.use(async (ctx, next) => {
  console.log(3)
  const axios = require('axios')
  const res = await axios.get('http://7yue.pro')
  next()
  console.log(4)
})

我们为什么一定要保证执行的顺序一定是洋葱模型？？ 我们举个例子： 我们在第一个中间件写一段代码逻辑，需要后面几个中间件执行完毕后才能生效，

如计时 计算后面所有中间件运行的总时长（这些逻辑是建立在洋葱模型下）

app.use(async (ctx, next) => {
  console.log(1)
  await next()
  console.log(2)
  // 计时函数
})

app.use(async (ctx, next) => {
  console.log(3)

如果不按照洋葱模型执行代码会出现什么样的情况？

这时候我们需要引入ctx 上下文的概念（之前提过koa中每一个中间件都会自动传入两个参数） 上下文有一个重要作用，在koa中基本都要操作ctx 上下文。

最常见的操作就是两个中间件或者多个中间件之间的传参，比如说我要把第二个中间件中axios获取到的结果传递到第一个中间件要怎么做呢？前面我们可以用return的方式，这样我们也能在第一个中间中接收到。

app.use(async (ctx, next) => {
  console.log(1)
  await next()
  console.log(2)
  // 计时函数
})

app.use(async (ctx, next) => {
  console.log(3)
  const axios = require('axios')
  const res = await axios.get('http://7yue.pro')
  return res
  next()
  console.log(4)
})

可是这有很大局限性的，如果几个中间都是自己编写的话，确实可以通过return的方式，可是koa十分精简，需要大量第三方的中间件。我们并不知道它们的执行顺序，这时候的传参就不能靠return了，需要靠ctx 上下文。

举个例子： 我们假设这两个中间件中加一个第三方中间件。

这种情况下我们怎么靠 ctx 上下文把第三个中间件的 res 传到第一个中间件呢？

每一个中间件函数执行的时候都会被注入ctx 上下文，这样我们就可以把要传的值在第三个中间件注入到ctx中，在第一个中间件中引入被注册到ctx的值。这里需要注意的是引入必须在第一个中间件的**next()**后面。这样才取得到值，如果在前面的话第三个中间件的线程都还没走完！！这里也直接的反应了洋葱模型的重要性。

app.use(async (ctx, next) => {
  await next()
  const r = ctx.r
  console.log(r)
  // 计时函数
})

app.use

app.use(async (ctx, next) => {
  const axios = require('axios')
  const res = await axios.get('http://7yue.pro')
  ctx.r = res
  next()
})

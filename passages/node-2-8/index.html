# 深入了解一下 async 和 await

首先从用法的角度上，async和await必须搭配在一起使用，不然就会报错。

# await

1.可以理解为求值关键字，可以结合上面的例子，如果不加 await，可以看到**next()**返回的是 Promise，加了 await 之后会对返回的 Promise 进行求值得到需要的值（一般我们会理解为等待的意思，当然其中有等待的意味，因为 await 会阻塞线程）。await 不仅仅只对 Promise 进行求值，还可以对表达式进行求值。 因为大多数的情况下返回的是 Promise，所以有了误解

我们这里试验一下：

app.use(async (ctx, next) => {
  console.log(1)
  const a = await (100 * 100)
  console.log(a)
  console.log(2)
})

2.await是可以阻塞当前的线程的，当await调用的是一个异步的函数，那么阻塞线程的意义就体现出来了

那么什么函数是异步的？ 对资源的操作（读文件，发送 HTTP 请求，操作数据库）

我们在next()（即第二个中间件）中发送一个http请求即是异步的，这里我们选择一个axios库(这个库最大的特点就是所有的返回都是基于 Promise)

提醒

这个库需要自行安装的 npm install axios，因为 koa 十分的精简

线程的阻塞并不是很好的表示出来,为了体现出线程阻塞的特点我们必须做一个假设，这时候我们假设这次异步操作是非常耗时的，比如我们访http://7yue.pro网速问题或者需要加载的内容很多。

我们通过看执行完方法的间隔时间，看有没有发生阻塞。

app.use((ctx, next) => {
  //   console.log(3)
  const axios = require('axios')
  const start = Date.now()
  const res = axios.get('http://7yue.pro')
  const end = Date.now()
  console.log(end - start)
  // next()
  //   console.log(4)
  //   return 'hello lemon'
})

我们可以看到时间间隔是很小的，是不是意味着请求是没有耗时的呢？

请求当然是耗时的，只是我们的代码的流程并不会等待请求的调用结果的返回。

下面我们看一下加上 await 是什么效果：

app.use(async (ctx, next) => {
  //   console.log(3)
  const axios = require('axios')
  const start = Date.now()
  const res = await axios.get('http://7yue.pro')
  const end = Date.now()
  console.log(end - start)
  // next()
  //   console.log(4)
  //   return 'hello lemon'
})

可以很明显的看到时间间隔变大了，显然当线程走到了 await 时会将线程阻塞不会走到下一步，等到 axios.get 取到结果并返回给 res，才会执行下一步。所以才看到时间间隔的变大.

 线程阻塞的意义非常重大，他可以把难以处理的异步函数变成同步的。
 而且 await 很强大，虽然说是阻塞线程，整个应用程序是不会卡住的，
 await 是阻塞了当前的线程，但是这个线程可以切换到其他地方运行其他代码

app.use(async (ctx, next) => {
  const axios = require('axios')
  const res = await axios.get('http://7yue.pro')
  console.log(res)
})

# 常见的前端传参方式

这一节我们聊一下Koa中的相关参数，我们前端需要经常向服务端传递参数的。

大概有四种我们最常见的传参方式：

  • url 路径里面传参
  • 在 url 的？后面进行传参
  • 在 http 中的 header 中传参
  • 在 http 中 body 进行传参

# postman 发送请求（携带参数）

我们暂时把classic.js 中的get改为post , 因为只有在post才能在http的body中进行传参。

router.post('/v1/:id/classic/latest', (ctx, next) => {
  ctx.body = { key: 'classic' }
})

以上就是我们最常用的四种形式的传递参数方式都覆盖了，现在我们点击postman的send就可以请求服务器的接口了。

# koa 获取参数

关键服务器如何获取到这四种传递过来的参数。

在 koa 里面可以使用 ctx.params 来取到 id 使用 ctx.request.query 来取到？问号后面的查询参数了 使用 ctx.request.header 来取到 header 里面的参数，由此可见这三种参数都很好获取。

在koa里面body里面的参数不是很好获取，我们需要通过一个库。

"koa-bodyparser":"^4.2.1"
1

通过使用这个库是可以获取到的,它实际上是一个中间件。因为它是一个中间件所以我们在app.js里面注册一下。

const parser = require('koa-bodyparser')
app.use(parser())
1
2

这样我们就可以在koa中使用bodyparser了，使用也非常的简单。使用ctx.request.body获取到body里面的参数。

classic.js整体代码：

const Router = require('koa-router')
const router = new Router()
router.post('/v1/:id/classic/latest', (ctx, next) => {
  const path = ctx.params
  const query = ctx.request.query
  const header = ctx.request.header
  const body = ctx.request.body
  ctx.body = { key: 'classic' }
})

module.exports = router

# 参数校验

接下来才是我们要讲的重点，我们需要校验参数 如果我们这些参数不能够很好的校验的话，我们写接口的时候将会非常的麻烦。 我们举个例子，比如我们的 id 号需要传一个正整数，可是用户没有传正整数传了一个字符串abc，那么我们是不是应该校验这种不合法的参数，并且返回给客户端。告诉用户这个参数传的不对。 第二个场景，有的参数是必须要传的，不能是空值。这时候也需要校验出来。 还可以举一个更偏向业务的例子，比如在用户注册的时候，他所填写的数据要符合 e-mail规范的，但是他传过来的是一个任意的字符串。这是必须校验并提示用户提交一个有效的数据。

校验是有两个层面的意义：

  • 防止非法的参数
  • 为客户端给出明确的提示，错在什么地方

这是前后端联调过程中非常重要的问题，很多时候开发效率低，很多时候是因为服务端人员在开发 api 的时候提示不够明确。

很蛋疼的是 koa 里面没有很好用的校验层，作者七月手撸了一个LinValidator

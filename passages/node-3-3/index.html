# api 兼容多版本

保留两个路由 v1,v2，然后我们讲一下 api 版本的概念，由第一个路由为例子返回的是 class，可是随着业务变动返回的值发生变化。**如果业务变更了,我们直接把代码修改了吗？**我们需要考虑客户端兼容性的问题。比如说某一个客户端是一个老版本，需要返回一个 class 我们发布的新版本需要返回其他的字段。如果客户没有将旧版本升级为老版本，这就会发生错误。

所以 api 需要兼容多个版本，那么我们该如何兼容和支持多个版本呢？

  • 把版本号加在 url 路径中去。
  • 把版本号加在查询参数中去
  • 把版本号加在 http 的 header 上面

这里有一个问题 我们是把两个路由版本都写在一个路由函数中还是分开写呢？

router.get('v1/classic/latest?version=v1', (ctx, next) => {
  ctx.body = { key: 'class' }
  if (v1) {
    //执行v1的逻辑
  }
  if (v2) {
    //执行v2的逻辑
  }
})

这么写可以吗？可以但是不好!

这么写会让代码的逻辑变得复杂，随着项目不断拓展，代码越来越多，版本也越来越多。可能版本会到v10，设置到v20。可以想像代码会有多长！而且如果我们的版本保存策略是兼容三个版本，那么旧的版本是需要删除的。比如说v1和v2版本我们是不要的。我直接把v1和v2的代码删除吗?这是一个很危险的行为。 因为修改代码是存在的风险的，很可能造成其他的代码也发生错误。

编程有一个很重要的原则：开闭原则
意思是说，我们在编程的时候对代码的修改是关闭的，对代码的扩展是开放的。

# 错误写法

所以我们应该为每一个版本的 api 写一个路由。先写一个 v1 版本的 api。在 api 文件下新建一个 v1 文件夹 api/v1 同理如果有 v2 版本，也在 api 文件夹下新建一个 v2 文件api/v2

那么假如说我们有两个api，第一个api是关于classic，第二个api是关于book。我们采取的版本号策略是把版本号加在 url 路径中去。其实就是我们上一节提到的两个主题，期刊和书籍。

const Koa = require('Koa')
const app = new Koa()
const Router = require('koa-router')
const router = new Router()
router.get('/v1/classic/latest', (ctx, next) => {
  ctx.body = { key: 'classic' }
})
router.get('/v1/book/latest', (ctx, next) => {
  ctx.body = { key: 'book' }
})
app.use(router.routes())
// API http://bl.7yue.pro/dev/index.html
app.listen(5000)

# 正确写法

正常的逻辑 高阶的模块可以导入低阶的模块 反之则不合适了。所以应该这么写。。。

修改完代码 重启一下服务 浏览器分别输入classic的api的url和book的api的url 刷新浏览器发送 http 请求 可以看到页面显示分别 {"key":"classic"} {"key":"book"}

最后我们抛出一个问题，现在只有两个路由 注册是比较简单的，在项目中往往有几十个路由。 而且每定义一个路由都要注册一下是不是挺麻烦的。有没有可以自动注册呢？
